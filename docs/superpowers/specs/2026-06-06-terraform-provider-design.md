# Terraform Provider for Nexspence — Design

**Date:** 2026-06-06
**Status:** Approved
**Deliverable:** `terraform-provider-nexspence` v0.1.0, published on registry.terraform.io

## Overview

A Terraform provider that manages Nexspence resources (repositories, blob stores,
users, roles, content selectors, privileges) through the existing REST API —
"instance as code". Analogous to `datadrivers/terraform-provider-nexus`, but
first-party and targeting Nexspence directly.

## Goals

- Full instance bootstrap from Terraform: blob store → repositories → content
  selector → privilege → role → user.
- Published on registry.terraform.io as `nexspence/nexspence` so users only need a
  `required_providers` block.
- Acceptance tests run against a real Nexspence instance — doubling as a
  Nexus-compat API regression suite.

## Non-Goals (v1)

- Cleanup policies, webhooks, replication rules, promotion rules, routing rules
  (deferred to v0.2+).
- Per-format-and-type resource types (`nexspence_repository_maven_hosted`, …) —
  one generic `nexspence_repository` instead.
- OpenTofu registry submission (can follow later; artifacts are compatible).
- Any changes to nexspence-core — the existing API and `AuthMiddleware` are
  sufficient.

## Repository & Stack

New **public** repo `nexspence/terraform-provider-nexspence`
(registry requires the `terraform-provider-<name>` naming), Go module
`github.com/nexspence/terraform-provider-nexspence`.

- **SDK:** `terraform-plugin-framework` (modern, type-safe; not the legacy
  `plugin-sdk/v2`). `terraform-plugin-testing` for acceptance tests.
- **API client:** thin hand-written REST client in `internal/client/`
  (patterns borrowed from the `nxs` CLI client, which is `internal/` and not
  importable). No OpenAPI codegen — not worth it for 6 resources.
- **Attribute naming:** follow `terraform-provider-nexus` conventions where they
  map, to ease migration for Nexus users.

```
terraform-provider-nexspence/
├── main.go                        # providerserver.Serve
├── internal/
│   ├── client/
│   │   ├── client.go              # base URL, auth, do() helper
│   │   ├── repositories.go        # /service/rest/v1/repositories/*
│   │   ├── security.go            # users, roles, privileges, content selectors
│   │   └── blobstores.go          # /service/rest/v1/blobstores/*
│   └── provider/
│       ├── provider.go
│       ├── repository_resource.go
│       ├── blobstore_resource.go
│       ├── user_resource.go
│       ├── role_resource.go
│       ├── content_selector_resource.go
│       ├── privilege_resource.go
│       ├── repository_data_source.go
│       └── repositories_data_source.go
├── examples/                      # rendered on the registry page
├── docs/                          # generated by tfplugindocs
├── .goreleaser.yml                # registry-compliant: GPG-signed SHA256SUMS
└── .github/workflows/             # ci.yml, release.yml
```

**API selection:** all resources call the Nexus-compat API (`/service/rest/v1/`) —
verified against `internal/api/router.go`: repositories
(`/repositories/:format/:type`), blob stores (`/blobstores/:type` — the compat
surface carries the full config), and all security objects
(`/security/users|roles|privileges|content-selectors`).

**Name ↔ ID resolution:** the Terraform schemas reference related objects by
**name** (human-friendly, matches HCL references), but the API relates them by
**ID** (`blobStoreId`, `contentSelectorId`, `privileges` as IDs, `roleIds`). The
provider resolves names → IDs on Create/Update (via list/get-by-name calls) and
maps IDs → names on Read.

## Provider Configuration

```hcl
provider "nexspence" {
  url   = "https://nexspence.example.com"
  token = var.nexspence_token        # nxs_* API token — preferred
  # or: username = "admin"  password = var.admin_password
}
```

- Env fallbacks: `NEXSPENCE_URL`, `NEXSPENCE_TOKEN`, `NEXSPENCE_USERNAME`,
  `NEXSPENCE_PASSWORD`.
- `token` and `username`/`password` are mutually exclusive; exactly one method
  required (validated in `Configure`).
- Auth goes through the existing `AuthMiddleware` (Bearer `nxs_*`, Basic, or JWT).

## Resources

### `nexspence_repository` (generic — one resource for all 14 formats × 3 types)

| Attribute | Type | Notes |
|---|---|---|
| `name` | string, required | ForceNew |
| `format` | string, required | enum of 14 formats; ForceNew |
| `type` | string, required | `hosted` \| `proxy` \| `group`; ForceNew |
| `blob_store` | string, optional | blob store name; default `default` |
| `allow_anonymous` | bool, optional | default `false` |
| `cleanup_policy_ids` | list(string), optional | |
| `quota_bytes` | number, optional | must be ≤ owning blob store quota (server-validated) |
| `proxy` | block | required iff `type = "proxy"`: `remote_url` (required). No remote auth attributes — the core `repoproxy` only supports `remote_url` today; add them in v0.2+ when core grows remote-auth support |
| `group` | block | required iff `type = "group"`: `member_names` (ordered list, min 1), `writable_member` (optional) |

Validators enforce block/type correspondence (`proxy` block forbidden unless
`type = "proxy"`, etc.).

### `nexspence_blobstore`

`name` (ForceNew), `type` (`local` \| `s3`, ForceNew), `path` (local),
`s3` block (`endpoint`, `bucket`, `region`, `access_key`, `secret_key`
sensitive), `quota_bytes`.

### `nexspence_user`

`username` (ForceNew), `password` (sensitive, write-only), `email`,
`first_name`, `last_name`, `enabled` (default true), `roles` (set of role names).

### `nexspence_role`

`name` (ForceNew), `description`, `privileges` (set of privilege names).

### `nexspence_content_selector`

`name` (ForceNew), `description`, `expression` (CEL).

### `nexspence_privilege`

`name` (ForceNew), `description`, `content_selector` (name). The `type` is always
`repository-content-selector` — hardcoded, matching the Nexspence privilege model.

## Data Sources (v1)

- `nexspence_repository` — look up one repository by name.
- `nexspence_repositories` — list all repositories.

## Cross-Cutting Semantics

- **Import:** `terraform import` by name for all six resources.
- **Secrets** (`password`, `secret_key`): `Sensitive: true`,
  write-only — the API never returns them, so Read does not diff them; changing
  the value in config triggers an Update.
- **External deletion:** Read receiving 404 removes the resource from state
  (standard drift semantics).
- **Errors:** API error bodies surfaced verbatim in diagnostics with the failing
  endpoint for debuggability.

## Testing

1. **Unit** — `internal/client` against `httptest.Server` mocks (CRUD per
   resource, error mapping, both auth modes); schema/validator tests. Run on
   every push.
2. **Acceptance** — `terraform-plugin-testing` with `TF_ACC=1` against a live
   stack (`ghcr.io/nexspence/nexspence` + postgres via docker compose). Per
   resource: Create → Read → Update → Import → external-delete drift → Destroy.
   Local entry point: `make testacc`.

## CI / Release

- **`ci.yml`:** `lint` (golangci-lint) → `test` (unit) → `acceptance`
  (compose stack) → `docs-check` (`tfplugindocs` output is committed and current).
- **`release.yml`:** tag `vX.Y.Z` → goreleaser → binaries
  (linux/darwin/windows × amd64/arm64), `SHA256SUMS`, GPG-signed sums,
  `terraform-registry-manifest.json` → GitHub release. The registry ingests new
  tags automatically after the one-time setup.
- **One-time manual setup:** generate GPG key → repo secrets `GPG_PRIVATE_KEY` /
  `PASSPHRASE` → sign in to registry.terraform.io via GitHub → register the
  public key → publish the provider.
- Versioning: semver, start at **v0.1.0**.

## Definition of Done (v1)

- All 6 resources + 2 data sources pass acceptance tests against the current
  Nexspence image.
- `terraform init` resolves `nexspence/nexspence` from registry.terraform.io.
- README contains the full bootstrap example (blob store → repos → RBAC → user).
- Provider mentioned in nexspence-core README + website (doc-sync rule).
