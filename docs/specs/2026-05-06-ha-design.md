# Phase 53: High Availability — Clustered Deployment

**Date:** 2026-05-06  
**Status:** Approved

## Goal

Enable Nexspence to run as a multi-node cluster behind a load balancer (both Kubernetes and Docker Compose), with shared PostgreSQL, Redis, and S3-compatible blob storage.

## Decisions

| Topic | Decision |
|---|---|
| DockerV2Auth cache | Move to Redis (TTL 30s) |
| Distributed locking | SetNX+TTL via Redis (single-node) |
| Metrics (persistent) | Remove in-process counters; read from DB on demand |
| Metrics (session) | Keep in-process per-node (`RequestsTotal`, `RequestErrors`) |
| HA example | Separate `docker-compose.ha.yml` |
| Single-node compat | `redis.enabled: false` degrades gracefully — no Redis required |

## Section 1 — Redis Client & Config

### New package: `internal/redisclient/`

Thin wrapper over `github.com/redis/go-redis/v9`. No global state — client is created in `main.go` and injected where needed.

```go
// client.go
type Client struct{ rdb *redis.Client }

func New(cfg Config) (*Client, error)
func (c *Client) Get(ctx, key) (string, error)
func (c *Client) Set(ctx, key, value string, ttl time.Duration) error
func (c *Client) SetNX(ctx, key, value string, ttl time.Duration) (bool, error)
func (c *Client) Del(ctx, key string) error
func (c *Client) Ping(ctx) error
```

### Config addition (`config.yaml`)

```yaml
redis:
  enabled: false          # false = single-node mode, no Redis required
  addr: "localhost:6379"
  password: ""
  db: 0
```

### Graceful degradation

When `redis.enabled: false`:
- DockerV2Auth cache → existing in-process `atomic.Bool` fallback
- Distributed locks → `NoopLocker` (always acquires immediately)
- `/readyz` → omits `redis` field from response

**Files:** `internal/redisclient/client.go`, `internal/config/config.go`, `config.yaml`, `config.yaml.example`

## Section 2 — DockerV2Auth Cache → Redis

### Current state

`handlers/auth.go` — `DockerV2Auth` uses closure-local `atomic.Bool` + `atomic.Int64` with 30s TTL. Node-local, diverges in multi-node setup.

### New behavior

```
GET nexspence:docker:anon_allowed
  hit  → use cached value ("1" or "0")
  miss → SELECT EXISTS(... format='docker' AND allow_anonymous=true)
         SET nexspence:docker:anon_allowed <value> EX 30
```

Redis key namespace: `nexspence:` prefix throughout to avoid collisions.

### Fallback

When `redisClient == nil` (redis disabled) → existing atomic cache logic unchanged.

**Files:** `internal/api/handlers/auth.go`

## Section 3 — Distributed Locking (Redlock)

### New package: `internal/distlock/`

```go
// lock.go
type Locker interface {
    Acquire(ctx context.Context, key string, ttl time.Duration) (Lock, error)
}

type Lock interface {
    Release(ctx context.Context) error
}
```

Two implementations:

| Implementation | When used |
|---|---|
| `RedisLocker` | `redis.enabled: true` — distributed lock via SetNX+TTL (single Redis node) |
| `NoopLocker` | `redis.enabled: false` — always succeeds, single-node safe |

Key naming: `nexspence:lock:<name>`

### Applied in

| Service | Lock key | TTL |
|---|---|---|
| `CleanupService.RunAll()` | `nexspence:lock:cleanup:run` | 30 min |
| `BlobStoreMigrationService.Start()` | `nexspence:lock:blobmig:<repoName>` | 2 hours |

`ScanService.trivyMu` stays as `sync.Mutex` — Trivy runs node-local, no cross-node conflict.

**Files:** `internal/distlock/lock.go`, `internal/distlock/redis.go`, `internal/distlock/noop.go`, `internal/service/cleanup_service.go`, `internal/service/blob_store_migration_service.go`

## Section 4 — Health Endpoints

Registered without auth middleware, before all other routes.

### `GET /healthz` — liveness

Always returns `200` if the process is alive. No external checks.

```json
{"status": "ok"}
```

### `GET /readyz` — readiness

Runs checks in parallel with 3s timeout each. Returns `503` if any check fails — load balancer / k8s stops routing traffic to the node.

```json
{
  "status": "ok",
  "checks": {
    "db":    "ok",
    "redis": "ok"
  }
}
```

- `db` — `SELECT 1` against PostgreSQL
- `redis` — `PING` against Redis (field omitted when `redis.enabled: false`)
- Any `"error"` value → top-level `"status": "degraded"` + HTTP 503

**Files:** `internal/api/handlers/health.go`, `internal/api/router.go`

## Section 5 — Metrics Refactoring

### Removed in-process persistent counters

Remove from `metrics.go`: `ArtifactsStored`, `BytesStored`, `DownloadsTotal`, `ArtifactsDeleted`.

These are DB-backed and diverge between nodes. The DB is already the source of truth.

Remove from `main.go`: the startup seed query that populated these counters.

### Replaced by on-demand DB query

When the system dashboard is requested, `SystemHandler` runs:

```sql
SELECT COUNT(*), COALESCE(SUM(size_bytes), 0), COALESCE(SUM(download_count), 0)
FROM assets
```

Result is returned directly — no caching needed (dashboard is a low-frequency admin call).

### Kept in-process (per-node, session-only)

- `RequestsTotal` — HTTP request count for this node's session
- `RequestErrors` — HTTP error count for this node's session

**Files:** `internal/metrics/metrics.go`, `cmd/server/main.go`, `internal/api/handlers/system.go`

## Section 6 — `docker-compose.ha.yml`

Minimal HA cluster for local testing and as a reference deployment.

### Topology

```
nginx (port 8081)
  └── upstream: nexspence_1, nexspence_2 (least_conn + /healthz check)

nexspence_1, nexspence_2
  ├── postgres  (shared DB)
  ├── redis     (shared — single-node sufficient for demo)
  └── minio     (shared S3 blob store)
```

### Services

| Service | Notes |
|---|---|
| `postgres` | Same as main compose |
| `redis` | `redis:7-alpine`, port 6379 (internal only) |
| `minio` | Same as main compose, ports 9000/9001 |
| `nexspence_1` | `redis.enabled: true`, `storage.type: s3` |
| `nexspence_2` | Same config as `nexspence_1` |
| `nginx` | `nginx:alpine`, exposes port 8081, round-robin with least_conn |

Both Nexspence instances: `NEXSPENCE_REDIS_ENABLED=true`, `NEXSPENCE_REDIS_ADDR=redis:6379`, `NEXSPENCE_STORAGE_TYPE=s3`.

### `nginx.conf` (embedded in compose or `deploy/nginx-ha.conf`)

```nginx
upstream nexspence {
    least_conn;
    server nexspence_1:8081;
    server nexspence_2:8081;
}
server {
    listen 8081;
    location / { proxy_pass http://nexspence; }
}
```

### `docs/ha-setup.md`

Covers: requirements, how to run the HA example, production Redis (Sentinel/Cluster) notes, k8s deployment hints (Deployment replicas, liveness/readiness probe config).

**Files:** `docker-compose.ha.yml`, `deploy/nginx-ha.conf`, `docs/ha-setup.md`
